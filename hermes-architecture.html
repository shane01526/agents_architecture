用三條思路理解 Hermes,自我學習是它的靈魂

Hermes Agent 是 Nous Research 打造的 self-improving(自我改進)AI 助理框架(Python)。它自稱「唯一內建學習迴圈的 agent」── 從經驗中創建技能、在使用中改進技能、提醒自己持久化知識、搜尋自己過去的對話、並跨 session 加深對你的認識。本文件基於 Hermes 開源 repo 的 website/docs/** 與真實原始碼逐一研讀整理,每個主張皆標註來源。

一句話心智模型:Hermes = 一個「會自己變好」的 agent runtime。它跑在一個常駐 gateway 上、能在 6 種 terminal backend 執行(含 Daytona/Modal serverless 休眠)、從多個訊息平台接觸你,而其每一次對話都可能在背景把學到的東西寫回記憶與技能庫,讓下一次更聰明。

★ 自我學習迴圈(Self-Improving Loop)

Hermes 的招牌:一個純 runtime 的閉環,讓 agent 從每次對話中累積記憶與技能,越用越貼合你 ── 全程不動模型權重。下面先給閉環總圖,再逐機制拆解到判斷依據與設計思維。

先講結論:這不是「模型訓練」runtime 自學 = 記憶 + 技能 + 設定 的累積(寫磁碟/DB),零權重更新、零梯度。Hermes repo 另有 batch_runner.py / trajectory_compressor.py 做 trajectory 生成與壓縮,用來離線訓練下一代 tool-calling 模型 ── 那是研究基礎設施,與此 runtime 閉環是兩回事(見 Security/RL)。

實作架構:它不是新子系統,而是「協調器 + 重用既有元件」

核心觀念:用 agent 跑 agentHermes 沒有為 self-improving 另寫一整套獨立子系統。真正算「新增」的只有一層薄協調器 agent/background_review.py ── 它在對話收尾時再 spawn 一個同樣的 AIAgent(但限制成只能用 memory + skills 工具),讓它回顧對話,然後透過既有的記憶工具、技能工具、session DB 把學到的東西寫回去。寫記憶用原本的 memory 工具、改技能用原本的 skill 工具、搜尋用原本的 session DB —— self-improving 是把這些既有零件「在每次對話收尾自動再跑一輪」的編排。

觸發判斷 ── 誰決定要不要 review

主迴圈每跑完一個 turn,把兩個計數器 +1:_iters_since_skill、_turns_since_memory。收尾時檢查三個條件同時成立才觸發:① 計數器 ≥ interval(預設 10)── 不是每 turn 都學,週期性回顧;② 這個 turn 沒被使用者中斷(/stop、打斷)── 半截對話學了會學到雜訊;③ 有產生 final 回應,且對應工具可用(要 review 技能就得有 skill 工具)。

💡 為什麼:把學習掛在「乾淨完成的 turn」收尾,而不是隨時跑 ── 既確保有完整素材可回顧,又不阻塞使用者的即時回覆(回覆早已送出,review 是事後背景做)。達標後計數器歸零,重新累積。→ Agent Loop

2

_spawn_background_review() ── fork 一個受限 agent(機制的心臟)

它不自己分析對話,而是 new 一個 AIAgent 實例(跟主對話同一個 class、同一套程式),但套上四層「緊身衣」:

① max_iterations=16:這個 sub-agent 自己也是個 agent loop(會思考、呼叫工具、看結果、再思考),16 是它的回合預算上限。為什麼要限:它背景跑、沒人盯,若不設上限萬一陷入「再查一下、再改一個」會無限燒 token;16 足夠它「回顧→決定→呼叫幾次工具→收尾」又不失控。

② 只開 memory + skills toolset:主對話 agent 可能有 terminal/browser/web/code 一大堆工具,fork 出來的 review agent 只給 memory 工具和 skill 工具,其餘全拿掉。為什麼:安全 + 聚焦 ── 它的任務只有「把學到的存起來」,不該也不能在背景偷跑 shell、開瀏覽器、發訊息;把工具白名單縮到最小,等於從機制上保證它只做記憶/技能這件事。

③ 繼承 parent memory store(_memory_store):它不是從磁碟重讀一份記憶,而是直接共用主對話 agent 的記憶物件。為什麼:它要看「現在已經記了什麼」才能判斷「這條新的該不該存、會不會跟舊的重複」;共用同一個 store,寫進去能立刻反映、不跟主 agent 狀態打架。

④ 暖快取 + 子層不再級聯:fork 通常繼承 parent 已建好的 system prompt(暖快取,省錢);而且它自己的 nudge interval 被設成 0 ── review agent 不會再觸發另一個 review(否則遞迴爆炸)。

3

兩段 prompt ── 告訴 fork「要找什麼」

fork 拿到的不是空白指令,而是一段精心寫好的回顧指令;依這次是 memory review 還是 skill review,餵不同的 prompt。

_MEMORY_REVIEW_PROMPT(記憶面,大意)「回顧上面的對話,考慮要不要存進記憶。聚焦兩點 ── ① 使用者透露了關於他自己的事(人設、偏好、個人細節)嗎?② 表達了對你該怎麼表現的期望嗎?有就用 memory 工具存;沒有就說『Nothing to save.』然後停。」

設計重點:明確區分「使用者是誰」(存 USER.md)和一般事實,並給「沒東西就明講並停」的出口,避免硬擠出無意義記憶。

_SKILL_REVIEW_PROMPT(技能面,核心是一條偏好序 ── 挑第一個符合的動作)① patch 一個本次對話載入過的技能(最優先)→ ② patch 一個既有的 class-level umbrella 技能 → ③ 在既有 umbrella 下加 references/ templates/ scripts/ 支援檔 → ④ 真的都沒有,才新建一個 class-level 技能。並明列「該捕捉」(使用者糾正你的風格/流程、非顯然的修法、載入的技能過時)vs「不該捕捉」(環境性失敗、對工具的負面斷言、一次性 transient 錯誤);新技能名必須是 class 級,不能用 PR 號或今天的任務碼。

設計重點:這條偏好序是為了避免技能庫長成一堆一次性窄技能,逼它優先「改進既有的」而非「狂建新的」,維持成少數厚實、可重用的 umbrella 技能。→ Skills

4

fork 自己跑迴圈、呼叫工具寫磁碟

拿到 prompt 後,這個 sub-agent 就像正常 agent 一樣跑:讀對話 → 推理「這次有什麼值得存」→ 呼叫工具。它呼叫的是既有工具:

• memory 工具(tools/memory_tool.py 的 MemoryStore)→ 寫 ~/.hermes/memories/MEMORY.md / USER.md(add/replace/remove)。→ Memory
• skill 工具(skill_commands.py 等)→ create/patch/write_file 技能檔。

💡 provenance:寫技能時,若是這個背景 review 建的新技能,會被 mark_agent_created() 標記 ── 這個標記讓之後的 curator 只整理「agent 自己建的」技能,不去動使用者手動裝的或內建的。

5

_digest_history ── 跨模型時的冷快取優化

正常情況 review agent 用跟主對話同一個模型,可以暖快取重放整段歷史。但若你設定讓 review 走不同(較便宜)的模型,那個模型沒有暖快取,重放整段長對話會很貴。

_digest_history 就是為此:把較舊的對話壓成一個精簡摘要(保留最近的尾段逐字),只送這個 digest 給 review agent,而不是整段原文。💡 為什麼:讓「用便宜模型做背景 review」在 token 上划算 ── 冷模型只需吃濃縮版就能完成回顧。

6

summarize_background_review_actions ── 把成果回報給你

fork 跑完後,它做了哪些動作(存了什麼記憶、改了哪個技能)會被這個函式整理成一句精簡摘要,丟回主對話顯示,例如:

💾 Self-improvement review: saved 1 memory, updated 2 skills

💡 為什麼:背景默默改你的記憶/技能會讓人不安。給一行可見的回報,讓你知道「它剛剛學了什麼」,維持透明與可控 ── 它是在你的對話裡學,不是黑箱。

7

Curator ── 另一條時間軸(非每 turn)

上面 1–6 是「每 ~10 turn 跑一次」的即時學習。Curator 是另一個週期更長的維護器(agent/curator.py):

• 何時跑:閒置 ≥2 小時 且 距上次 ≥7 天(should_run_now() 判斷),不是每 turn。
• 做什麼:① 確定性轉換(不用 LLM)── 技能 30 天沒用標 stale、90 天沒用移到 .archive/;② 可選的 LLM 整併(consolidate,預設關)── 讀 agent 自創的技能,把近重複的合併成 umbrella;③ 每次動手前先備份(可 rollback)。

💡 為什麼分開:即時 review 負責「長」(把新學的加進來),curator 負責「修剪」(把舊的、重複的清掉)── 一個高頻一個低頻,合起來讓技能庫既會成長又不會無限膨脹變垃圾場。→ Cron

一句話收束整條流程的精髓:「在乾淨 turn 收尾,fork 一個工具受限的 sub-agent,餵它一段教它『該存什麼』的 prompt,讓它用既有的 memory/skill 工具把學到的寫回磁碟,再回報一行給你;另由低頻的 curator 定期修剪。」 沒有任何一步在動模型權重 ── 全是用既有零件組出來的編排。流程為 repo 2026 快照研讀整理,精確變數名/行為若要正式引用建議再對一次 repo。

隔離粒度:這個 review 替「誰」學?

很多人會擔心:多個 agent、多個使用者共用一套系統,那「學到的東西」會不會混在一起?要分兩個軸來看,答案不一樣 ──(A)不同主 agent (B)不同使用者。

✅ (A) 不同主 agent(profile)= 完全隔離Hermes 用 profile 區分不同 agent(hermes -p <name>),每個 profile 有自己的家目錄:獨立的 SOUL.md、MEMORY.md/USER.md、skills/、state.db。background review fork 的關鍵設計是「繼承 parent 的 memory store」── 它寫回的,就是當下那個主 agent 所屬 profile 的記憶與技能庫。所以 coder profile 的對話只會改進 coder、writer 改進 writer,兩者天然分開。
⚠️ (B) 不同使用者 = 預設「不分」一個 profile 可能同時服務多個平台使用者(Telegram 的 A、Discord 的 B…),但內建記憶是 profile 級的單一檔案 ── MEMORY.md + USER.md 不會自動按使用者切成多份,多人共用同一 profile 時偏好會混進同一份;技能庫也是 profile 級、所有使用者共享;跨 session 回憶(FTS5)搜的是整個 profile 的 state.db。review fork 是從當下 session spawn、繼承的是 profile 級 store,不是 per-user。本質上是「單一可信使用者」導向(呼應 OpenClaw「非多租戶」的立場)。

💡 為何這樣OpenClaw 以清晰的信任邊界 + 標準威脅模型界定安全;Hermes 因背後是研究機構(Nous),把 runtime 產出的 trajectory 拿來離線訓練模型—— 這是「動權重」之處,與 runtime 自學分開。

1為什麼存在

3運作機制

  • 訊息交替規則:system → user/assistant 交替;工具呼叫時 assistant(含 tool_calls)→ tool → … → assistant;只有 tool role 可連續。

  • iteration 預算:預設 90(agent.max_turns),子代理獨立預算(上限 delegation.max_iterations=50);100% 時回工作摘要。
  • fallback 鏈:429/5xx/401/403 → 試 fallback provider(401/403 先試刷新憑證);in-place 換 model/provider/client。
  • callback 面:tool_progress / thinking / reasoning / clarify / step / stream_delta / status 等。

2核心概念

  • API-call-time-only 層(不進快取 prompt):ephemeral_system_prompt、prefill、gateway session 疊加、後期 Honcho recall、pre_llm_call plugin context。
  • 平台提示可自訂:platform_hints.<platform>.{append|replace}。

Delegation & Kanban

上游
delegate_task 工具
本元件
子代理並行(同進程)+ Kanban 多進程 worker lanes
下游
各自 terminal session

1為什麼存在

複雜任務要並行多工作流而不污染主 context;長期/跨進程協作則需要更耐久、可審計的看板,而非脆弱的 in-process swarm。

2核心概念

兩個層級delegate_task:同 gateway 內 spawn 子 AIAgent(隔離 context、受限 toolset、各自 terminal),只回最終摘要進父 context。Kanban:SQLite 看板(~/.hermes/kanban.db),每個 worker 是獨立 OS 進程(hermes -p <assignee> chat -q)。

3運作機制

  • delegation:預設最多 3 並行子代理;巢狀需 role="orchestrator"(max_spawn_depth 預設 1=扁平);葉子不能再 delegate/clarify/memory/send_message/execute_code;中斷父則中斷所有子;子繼承父的 key/provider/credential pool;可用便宜模型(delegation.model)。
  • Kanban worker lane:assignee(profile 名)+ spawn 機制(注入 HERMES_KANBAN_* 環境變數)+ 生命週期終止(kanban_complete / kanban_block / 進程退出);review-required block 模式;stale claim TTL(15min)、crash 偵測、consecutive-failure breaker、max_runtime。

Kanban 深入:多進程協作看板

Kanban 是一個 SQLite 支撐的任務看板,讓多個 Hermes agent(各自是獨立 OS 進程)像團隊一樣協作 ── 每個任務是一列、每次交接也是一列、每個 worker 是有自己身分的完整進程。它要取代的是「在同一進程裡硬塞一堆 subagent」那種脆弱做法。

跟 delegate_task 差在哪(為什麼要有它)

面向delegate_task(子代理)Kanban
worker 是什麼同進程裡 fork 的子 agent獨立 OS 進程(hermes -p <name>)
持久性在記憶體、隨主進程消失存 SQLite,跨重啟存活
人能介入嗎不太行能(CLI / dashboard 隨時看與改)
適合短暫並行子任務、回個摘要長期、可審計、需人介入的協作

能做到什麼

多 agent 並行協作

不同 profile(可不同人格/技能組)各認領各的卡,真正平行跑。

任務依賴與交接

kanban_link 串任務、block/unblock 表達「等別人先做完」;交接都是看板上一列,人或 agent 都能讀寫。

人機混合工作流

人開卡、agent 做、人審核 ── 像用 Trello/Jira,但執行者是 AI。

長時間、跨重啟

狀態在 SQLite,gateway 重啟、worker 換機都不丟。

完整稽核軌跡

每任務有 stdout/stderr log、task_runs(每次執行 exit code/summary)、task_events(promoted/claimed/heartbeat/completed/blocked/crashed…);hermes kanban tail <id> 即時、runs <id> 歷史。

優勢(為什麼這設計好)

① 進程級隔離 = 穩每個 worker 是獨立 OS 進程、有自己身分。一個 worker 崩了不會拖垮別人或主 gateway ── 比「in-process subagent 群」健壯太多(這正是它的設計動機)。
② 耐久 + 可恢復(內建四項防呆)stale claim TTL(預設 15 分鐘):認領卻不 heartbeat/完成的 worker,確認進程已死才回收(活著的會 extend 而非被搶);crash 偵測 + consecutive-failure 斷路器:一直失敗的任務自動 block,不無限重跑燒資源;per-task max runtime:硬上限防單一任務跑到天荒地老;stranded 偵測:ready 卻長期沒人認領的卡在 hermes kanban diagnostics 報警。
③ 可觀測、可審計每一步都是看板上的列 + log,出事可回溯;不是黑箱 swarm。
④ 人能隨時介入review-required、comment、unblock ── 把「人類在迴圈」做進機制,適合需要把關的正式工作。
⑤ 天然對接多 profileworker 就是 hermes -p <profile>,可用不同 profile 當不同「角色工人」(如 coder、reviewer、researcher),各有各的技能/人格。→ Profile
一個具體想像你開三張卡:研究競品(指派 researcher)、寫實作(指派 coder,link 依賴前一張)、審查(review-required)。dispatcher 各自 spawn 一個 hermes -p … 進程;coder 做完不 complete 而是 block 成「待審」;你在 dashboard 看 diff、留 comment、unblock → coder 進程帶著你的意見重新跑修。全程狀態在 SQLite,中途關機再開都接得回來。env 變數名/TTL 預設值為 repo 2026 快照研讀整理,正式引用前建議再對一次 source。

Security & RL 研究角度

上游
backend 隔離
本元件
危險指令核准 · 威脅掃描 · 網路 egress 隔離
約束對象
exec/工具memory 內容

1為什麼存在

能跑 shell、開瀏覽器、裝 MCP 的 agent 既強大也危險。需要明確的核准閘、內容掃描與網路隔離,讓強大與安全可取捨。

2核心概念

多道防線① DANGEROUS_PATTERNS 核准(rm -rf / mkfs / DROP TABLE / curl|sh…)② 威脅掃描(memory/context/SOUL 載入與寫入,中毒換 [BLOCKED])③ 網路 egress 隔離(docs/security/network-egress-isolation.md)④ backend 隔離(docker/modal/daytona)。

3運作機制

  • 核准流:CLI 互動 / gateway 非同步回呼;可選 LLM 智能自動核准低風險;一次核准本 session 不再問;allow permanently 寫 config。
  • 憑證隔離:provider key 只送對應 host(OPENROUTER_API_KEY 只給 openrouter.ai)。
  • memory 掃描 scope strict(因進 frozen prompt,一則中毒污染整個 session)。
