用三條思路理解 OpenClaw

OpenClaw 是一個個人 AI 助理框架:跑在你自己的裝置上,透過你已經在用的訊息軟體回應你。它源自一個學習 AI 的個人實驗,歷經數次更名(Warelay → Clawdbot → Moltbot → OpenClaw)。本文件基於 OpenClaw 開源 repo 的 docs/、src/、extensions/ 與部署檔逐一研讀整理,每個重要主張皆標註來源檔案路徑。

一句話心智模型:Gateway 是 agent 的「家」 ── 一個常駐的單一控制平面(control plane),擁有所有 session、channel 連線、認證與狀態;所有客戶端(CLI、Web UI、桌面 App、手機/headless 節點)都透過同一條 WebSocket 連到它。其餘所有子系統都圍繞 Gateway 運作。

為什麼用「三條路徑」而不是一份清單?

上一版本把 10 個主題平行排列、各自陳述事實 ── 正確,但讀起來像參考手冊的傾倒,缺乏「為什麼這樣設計」的因果線。理解一個 agent 框架其實需要三種不同的思路,而它們指向同一組元件:

🚶

路徑一 · 訊息的旅程

想知道「執行時實際發生什麼」

跟著一則訊息,從進站、路由、排隊、組裝脈絡、呼叫模型、跑工具、串流回覆,一路到寫回記憶與背景延續。動態、線性、有故事性。

🏗️

路徑二 · 由下而上

想建立「為什麼這樣分層」的直覺

從最底層的棲身環境開始往上疊:住哪 → 它是誰 → 怎麼想 → 記得什麼 → 能做什麼 → 怎麼對外 → 怎麼自主與被約束。靜態、結構性。

路徑一 · 動態視角

🚶 一則訊息的旅程

這條路徑跟著一則訊息從進來到回覆、再到背景延續的完整生命週期。每一站都標明它做什麼、為什麼重要,並提供「深入 →」連到對應的元件詳解。讀完這條路徑,你會理解 OpenClaw 在執行時實際發生了什麼。

舞台早已就緒在第一則訊息抵達之前,Gateway 早已是一個常駐 process:它綁在 127.0.0.1:18789、維持著所有 channel 的連線(WhatsApp 的 Baileys session、Telegram 的 grammY 連線…)、載入好 plugin 與 skills 快照。旅程的每一步都發生在這個常駐的家裡。→ 棲身/Gateway
1

進站 · Channel adapter 把平台事件正規化

訊息先抵達對應的 channel adapter(每個平台一個 plugin)。Adapter 的工作是把平台原生事件翻成 OpenClaw 的正規化 inbound 訊息。此時做兩件防護:去重(短期 cache,防止斷線重連造成的重送再次觸發執行)與 debounce(快速連發的同人文字訊息可合併成一個 turn;媒體/附件即時 flush;控制指令繞過 debounce)。Body 會分離成 BodyForAgent(模型面向)、CommandBody(指令解析)等欄位。

深入 → Channels & Chat
2

路由 · 「最具體者勝」選定一個 agent 與 session

路由是確定性的、由主機設定控制 ── 模型從不選 channel 或 agent。系統依序嘗試 bindings 比對(精確 peer → parent peer/thread → guild+roles → guild → team → account → channel → 預設 agent),首個命中勝出。選定的 agent 決定了用哪個 workspace 與 session store。接著算出 session key:DM 收斂到 agent:<id>:main;群組/頻道按 channel 隔離成 agent:main:telegram:group:…:topic:… 之類。

深入 → 路由  深入 → 多代理 bindings
3

排隊 · session lane 序列化 + 檔案寫鎖

為什麼要排隊?因為每個 session 同時只能有一個活躍執行,否則工具與 session 歷史會競爭。OpenClaw 用 session lane(session:<key>)序列化,再進入全域 lane(預設 main,並行上限 4;subagent lane 為 8)。Transcript 寫入另受檔案層級 session write lock 保護(process-aware,連跨 process 的寫入都攔得到;預設等待上限 60000ms)。若此時又來新訊息,預設以 steer 模式接入當前執行。

深入 → 佇列/Lane/Steering  深入 → Session 寫鎖
4

Loop 啟動 · agent RPC → runEmbeddedAgent

agent RPC 驗證參數、解析 session、持久化 metadata,然後立即回傳 {runId, acceptedAt}(非阻斷 ── 呼叫者拿到 runId 就能去監聽串流或用 agent.wait 等待)。接著 agentCommand 解析模型與 thinking/verbose 預設、載入 skills 快照,呼叫 runEmbeddedAgent。後者序列化執行、解析 auth profile、建立 OpenClaw session、訂閱 runtime 事件,並設定逾時計時器。

深入 → Agent Loop
5

組裝脈絡 · context engine 決定模型看到什麼

這是 agent 品質的關鍵一步。Context engine 的 assemble 回傳符合 token budget 的有序訊息;系統把 workspace 的 bootstrap 檔案(AGENTS.md/SOUL.md/USER.md/MEMORY.md…)注入 system prompt 的 Project Context。若啟用 active memory,一個阻斷式記憶子代理會先跑,在主回覆前把相關長期記憶以隱藏前綴浮現出來。Memory 是磁碟上的 Markdown ── 模型只記得被存下來的東西,沒有隱藏狀態。

深入 → Context Engine  深入 → Memory & Active Memory
6

呼叫模型 · 解析 Provider / Model / Runtime

在送出前,OpenClaw 解析四個層次:provider(怎麼認證)、model(選哪顆模型)、agent runtime(哪個迴圈執行這個 turn ── 內建 openclaw、或 plugin harness 如 codex/copilot、或 CLI backend claude-cli)、channel(訊息進出處)。Runtime 選擇有明確優先序(model-scoped → provider-scoped → auto)。這一步決定了「誰擁有模型迴圈」。

深入 → Agent Runtime
7

工具迴圈 · 模型要動作 → 篩選 → 執行 → 回灌

模型若決定呼叫工具,OpenClaw 並非把所有工具都攤給它 ── 工具經8 層 policy 篩選(profile → provider → 全域 allow/deny → agent → sandbox → subagent…),每層只能進一步限制,絕不加回。模型只看到存活的工具的 schema。Skills(Markdown 指令)早在第 4 站就注入了 system prompt,教模型何時/如何用這些工具。工具執行的結果(經 size/image 清洗)回灌進對話,迴圈繼續直到模型給出最終回覆。

深入 → Tools 與 policy  深入 → Skills
8

串流回覆 · 確定性地回到原 channel

Assistant delta 串流出來。注意:到 channel 沒有真正的 token-delta 串流 ── 預覽是基於訊息的(送出 + 編輯)。期間有 typing 指示器、progress draft(單一持續更新的進度訊息)。靜默 token NO_REPLY 會被過濾掉。最終回覆確定性地回到訊息來的那個 channel(不是模型挑的)。

深入 → 串流/typing/progress
9

收尾 · 記憶刷新 → 寫 transcript →(可能)壓縮

執行寫進 append-only JSONL transcript。若 context 接近視窗上限,觸發 compaction(摘要較舊對話,完整歷史仍留磁碟)。壓縮之前會先跑一個靜默的「現在寫記憶」turn(回覆以 NO_REPLY 開頭,使用者看不到),提醒 agent 把重要脈絡存進記憶檔,避免壓縮造成脈絡流失。

深入 → Compaction  深入 → 記憶刷新
10

之後 · 背景延續(heartbeat / commitments / dreaming)

旅程沒有真正結束。Heartbeat 每 30 分鐘跑一次 main-session turn,批次檢查收件匣/行事曆。對話中推斷的 commitments(「明天面試後關心一下」)會在未來經 heartbeat 投遞。夜裡 dreaming(預設關閉)把短期素材鞏固、晉升到 MEMORY.md。這就是 OpenClaw 的「學習」── 記憶與設定的累積,不是權重微調。

路徑二 · 結構視角

🏗️ 由下而上建構心智模型

路徑一講「發生什麼」;這條路徑講「為什麼這樣分層」。我們從最底層往上疊,每一層先回答「為何需要這一層」,再說明它上面承載什麼。每層都連到對應的元件詳解。

L0 · 地基

棲身 Substrate + Gateway

為何需要這層:agent 要能「一直在線、隨時回應」,就必須有個常駐的東西擁有所有連線與狀態 ── 否則每次都要冷啟動、無法跨客戶端共享 session。

Gateway 是單一、常駐的控制平面(每主機一個),綁 127.0.0.1:18789,以一條多工 WebSocket 承載 RPC、HTTP API、Control UI、Canvas。Substrate 是你選擇跑它的地方:本機 PC、Raspberry Pi、VPS、雲端、Docker、K8s。其上才談得上其他一切。

深入 → 棲身/Gateway/Nodes
L1 · 身分

Identity + Workspace(它是誰)

為何需要這層:同一個 Gateway 可能服務多個用途/多個人,而模型每個 session 都是全新實例 ── 必須有檔案來定義「這個 agent 是誰、它的連續性在哪」。

每個 agent 有自己的 workspace(檔案工具的家)與身分檔:SOUL.md(聲音/語氣)、AGENTS.md(操作規則/standing orders)、USER.md、IDENTITY.md。多個 agent 可在一個 Gateway 上完全隔離,靠 bindings 路由。

深入 → Identity & Multi-Agent
L2 · 大腦

Runtime + Agent Loop(它怎麼想)

為何需要這層:有了家與身分,還需要一個確定性的引擎來驅動「intake → 推理 → 工具 → 回覆 → 持久化」的迴圈,並保證狀態一致(不能讓 LLM 直接改狀態)。

核心鐵律:Runtime(確定性控制)≠ 模型(非確定性推理)。Agent runtime 擁有準備好的模型迴圈;agent loop 是每 session 單一、序列化的真實執行,發出 lifecycle/assistant/tool 三條串流。Runtime 有家族(embedded openclaw/codex vs CLI backend)。

深入 → Runtime  深入 → Agent Loop
L3 · 記憶

Session · Memory · Context Engine(它記得什麼)

為何需要這層:模型 context 視窗有限且無狀態,但助理必須記得過去 ── 需要持久層(磁碟)、可插拔的檢索,以及把有限視窗填好的機制。

Session 是每對話的 JSONL transcript(短期);Memory 是磁碟上的 Markdown(長期 MEMORY.md + 每日筆記),可插拔後端;Context engine 控制每 turn 模型看到什麼,並在視窗滿時 compaction。三者合起來是「記憶」的全貌。

深入 → Session  深入 → Memory  深入 → Context/壓縮
L4 · 能力

Tools + Skills(它能做什麼)

為何需要這層:會想會記還不夠 ── agent 要能做事(跑指令、開瀏覽器、查網路、發訊息),也要被教會何時怎麼用這些能力,並受權限約束。

Tools 是型別化的可呼叫動作,經多層 policy 篩選後才攤給模型。Skills 是 Markdown 指令包,注入 prompt 教模型工作流程(skill ≠ tool)。tool search 讓大型 catalog 不必全部攤開。

深入 → Tools  深入 → Skills
L5 · 對外

Channels + 聊天前端(它怎麼接觸世界)

為何需要這層:助理的價值在於「在你已經用的地方回應你」── 需要把 20+ 訊息平台與多個前端統一接到同一個大腦,且路由可預測。

Channels 是訊息表面的 adapter(WhatsApp/Telegram/Slack/Discord…),把平台事件正規化、把輸出翻回原生 send。前端(WebChat/TUI/Control UI)都連同一條 Gateway WS。回覆確定性回到來源。

深入 → Channels & Chat
L6 · 自主與治理

Automation/Learning · Plugins · Security

為何需要這層:一個真正有用的助理要能主動(不只被動回覆)、能被擴充、且整個強大的系統要被約束在可信邊界內。

Automation(cron/heartbeat/standing orders/commitments/task flow)給它自主性;Learning = 記憶+設定累積 + dreaming 鞏固(非權重)。Plugins 是「一切皆 plugin」的擴充模型。Security 定義信任邊界(單一可信操作者、非多租戶)。

1為什麼存在

一個「個人 AI 助理」的核心承諾是一直在線、隨時在你用的地方回應你。要做到這點,必須有個東西:① 常駐不關;② 擁有所有 channel 連線(WhatsApp 一台主機只能開一個 session);③ 集中保管 session、認證、狀態,讓多個客戶端(手機 App、CLI、Web)共享同一個對話脈絡。如果沒有這層,你會得到一堆各自為政、無法協調、每次冷啟動的腳本。Gateway 就是為了當這個「家」而存在。

2核心概念

把 Gateway host 想成「agent 住的地方」它擁有 session、auth profile、channel、狀態。你的筆電、桌機、節點都連到那個主機。每主機一個 Gateway。docs/gateway/remote.md

Substrate 則是你選擇跑這個 Gateway process 的環境。Gateway 與 substrate 的關係,就像「程式」與「它跑的機器」。

3運作機制

Gateway:一個 process、一個多工 port

  • 一個常駐 process 負責路由、控制平面、所有 channel 連線。
  • 一個多工 port(預設 127.0.0.1:18789)承載全部:WS 控制/RPC、HTTP API(/v1/models /v1/chat/completions /tools/invoke)、plugin HTTP route、Control UI、Canvas host(/__openclaw__/canvas/)。
  • Bind 模式:loopback(預設)/lan/tailnet/custom/auto。非 loopback bind 預設要求認證。
  • 監督:macOS launchd(ai.openclaw.gateway)、Linux/WSL2 systemd user service、Windows 排程工作。

Substrate 矩陣與權衡

環境權衡
本機 macOS選單列 App 擁有 TCC 權限、管理 launchd、把 Mac 能力當 node 暴露。筆電睡眠 agent 也睡。勿放 iCloud 同步資料夾。
本機 Windows三條路:Windows Hub(WinUI、免管理員、tray)、原生 PowerShell Gateway(排程工作)、WSL2(最相容,需啟用 systemd)。
本機 LinuxGateway 完整支援;有 OOM 受害者偏置(子 process 先被殺,Gateway 存活)。
Raspberry Pi$35–80 一次性、無月費。只有 gateway 在本機,模型走雲端 API。勿在 Pi 跑本地 LLM。
VPS常駐、便宜、不睡。Gateway 是真相來源 ── 備份狀態+workspace。安全預設:保 loopback,經 SSH/Tailscale 連入。
雲端(Fly/Render/Railway/Hetzner/GCP/Azure/Oracle/DO)有真實部署檔(fly.toml、render.yaml)。需持久 volume + token + 非 loopback bind。Render 免費層 15min spin down 丟狀態。

macOS VM為 macOS-only 能力(iMessage)或嚴格隔離。建議仍以小型 Linux VPS 為主,hybrid = VPS Gateway + Mac 當 node。

Nodes:裝置伴生(不是 gateway)

Nodes 是周邊,不是 gatewayNode 以 role:"node" 連到 Gateway WS(同 port),暴露指令(canvas.* camera.* screen.record location.get system.run)。它不跑 gateway 服務。Channel 訊息落在 Gateway,絕不在 node。

指令流(Telegram → node):① 訊息到 Gateway → ② Gateway 跑 agent、決定呼叫 node 工具 → ③ 經 WS 呼叫 node → ④ node 回結果,Gateway 回 Telegram。兩道閘:node 須在 connect.commands 宣告 + Gateway 平台政策須允許(危險指令如 camera.snap 需明確 gateway.nodes.allowCommands)。

Network model 與 Sandbox

Loopback first;非 loopback 需有效認證(gateway.auth.mode:none/token/password/trusted-proxy)。遠端偏好 Tailscale/VPN > SSH tunnel。Sandboxing 可選且預設關閉,只隔離工具執行 ── Gateway 本身從不被 sandbox(詳見 Security、Tools)。

1為什麼存在

1為什麼存在

2核心概念

  • 單獨的 prompt injection 不是漏洞(須跨越 policy/auth/approval/sandbox/tool 邊界)。
  • 可信安裝的惡意 plugin 做特權事 = 已記載行為。
  • 共享 gateway 上的 per-user 隔離期待 = 範圍外。
  • Agent/模型不是可信 principal ── 安全來自主機/設定信任、auth、tool policy、sandbox、exec approvals。

威脅模型(MITRE ATLAS)

5 個信任邊界:① Channel Access(device 配對 1h DM/5m node grace、AllowFrom、token/password/Tailscale auth)② Session Isolation ③ Tool Execution(Docker sandbox 或 host exec-approvals;SSRF = DNS pinning+IP 封鎖)④ External Content(抓取內容包在 XML tag+告示)⑤ Supply Chain(ClawHub)。

P0 威脅與殘餘風險(誠實揭露)直接 prompt injection(「只偵測、不阻擋」)、惡意 skill 安裝、credential harvesting。Token 明文儲存;無 rate limiting。命名安全檔:src/gateway/auth.ts src/infra/net/ssrf.ts src/security/external-content.ts src/infra/exec-approvals.ts

認證:兩個不同的層

Gateway 連線 auth

gateway.auth.mode:none/token/password/trusted-proxy。Device 配對:首次需核准,loopback 自動核准,scope 升級需重新核准。

模型 provider auth

OAuth+API key per provider。Auth profile 現存於每 agent SQLite。OAuth profile 預設不可攜。

Secrets ── SecretRef

選用、附加式,讓憑證不必明文。三來源:env/file/exec。解析成記憶體 runtime 快照(啟動 eager、fail-fast、reload 原子交換 last-known-good)。整合:1Password/Bitwarden/Vault/pass/sops。

關鍵警告「SecretRef 不是 process 隔離邊界。」若明文仍在 agent 可讀檔案,agent 能用 file/shell 工具讀到。OAuth/輪替憑證刻意排除於 SecretRef 解析。

元件詳解 · 大腦

任務分配 & 並行控制(Lane 佇列)

上游
路由算 session key
本元件
session 分流 + 兩層 lane 佇列 + queue 模式
下游
序列化執行寫鎖

1為什麼存在

一個 gateway 要同時服務多 channel、多人、多群組。必須決定「誰能並行、誰要排隊」── 否則同一對話的訊息會競爭、transcript 會錯亂,或全部塞一條佇列拖垮吞吐。

2核心概念

session 是並行的基本單位路由先算 session key(DM→agent:<id>:main;群組→agent:<id>:<channel>:group:<id>)。不同 key = 可並行;同 key = 序列化。

3運作機制

  • 兩層 lane:session lane(session:<key>,每 session 同時僅一個活躍執行)+ 全域 lane(預設 main,並行上限 agents.defaults.maxConcurrent 預設 4;subagent lane 8、cron/cron-nested 另計)。
  • queue 模式(執行中又來新訊息):steer(預設,turn 邊界接入)/ followup / collect(靜默窗合併)/ interrupt(中止跑最新);debounceMs:500、cap:20、drop:summarize。
  • session write lock:檔案層級鎖(process-aware,等待上限 60000ms)再保證 transcript 單寫者。
  • 對應系統:src/agents/embedded-agent-runner/lanes.ts、src/process/command-queue.ts(lane-aware FIFO + 並行上限)。

3.5逐步推演:很多人同時打進來

關鍵是「先用 session key 分流,再決定並行或排隊」。

◇兩則訊息是同一個 session key 嗎?routing
不同 key各自進自己的 session lane → 可並行(總數受 main lane maxConcurrent=4 節流;超過則於全域 lane 排隊)
同一 keysession lane 序列化 ── 同時只有一個活躍執行
◇同 session 執行進行中又來新訊息?
steer(預設)turn 邊界把新訊息接進當前執行
collect/followup合併成下一 turn / 排到下一 turn
interrupt中止當前、改跑最新;cap 滿則 drop:summarize

4與其他元件的關係

連到鄰居上游:路由算 session key 決定分流。下游:每 session 由 Agent Loop 序列化執行、取 write lock;子代理走 subagent lane(Tools)。

🔎 Audit & Logs

🔎 稽核:留什麼 log 給人查核

OpenClaw 的 audit 是本機優先、操作者導向:所有對話與執行記錄留在本機、預設不外送,並提供一個可一鍵匯出、redacted 的稽核束專門給人查核或回報問題。

定位:可攜的稽核束最「為人查核」的設計是 /export-trajectory ── 一鍵把當次執行打包成 redacted bundle(含 system prompt、工具清單、事件流),可直接附在 issue 裡。其餘 transcript / gateway log / exec 核准記錄皆本機留存,要外送才接 OTel/Prometheus plugin。
